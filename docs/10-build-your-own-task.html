11  Translating Your Own Problem into PyTorch

If this book only leaves you able to understand the examples inside it, that is not enough.

The real goal is this:

when you face your own task later, you should be able to translate it into a PyTorch project.

This chapter is here to train exactly that skill.

11.1 Start with the General Template

Open code/08_project_template.py.

The code is intentionally plain. It is not trying to show clever tricks. It is meant to give you a starting point that you can actually reshape for your own work.

It contains four parts that you will almost always need to modify:

  1. YourDataset
  2. YourModel
  3. criterion
  4. train_one_epoch()

11.2 What Does Each Part of the Template Do?

11.2.1 YourDataset

This defines:

  • what your raw data looks like
  • what one sample should return

In other words, it solves the data entry problem.

11.2.2 YourModel

This defines:

  • which layers the input passes through
  • what format the final output has

In other words, it solves the model structure problem.

11.2.3 criterion

This defines:

  • what standard you use to judge whether the current prediction is good or bad

In other words, it defines the optimization target.

11.2.4 train_one_epoch()

This defines:

  • how parameters get updated repeatedly during one epoch of training

In other words, it defines the training process.

When you begin building your own task, most of the work is some variation of editing those four parts.

11.3 Every Task Should Start with These Four Questions

11.3.1 1. What is your input?

Is it images, text, tabular data, or a time series?

That decision affects:

  • how to write the Dataset
  • what the input tensor shape should be
  • what preprocessing is needed

11.3.2 2. What is your output?

Is it a continuous value, one category, or multiple labels?

That decision affects:

  • the design of the final output layer
  • the choice of loss function

11.3.3 3. How do you define success?

This determines which metrics matter:

  • regression often uses error-based metrics
  • classification often uses accuracy, recall, or similar metrics
  • ranking and generation tasks use different metrics again

11.3.4 4. What training skeleton will you use?

This affects:

  • how batching works
  • which optimizer you choose
  • whether you need a validation set
  • whether you should save the best model

11.4 Turn Those Questions into Concrete Decisions

Many people still get stuck here. They know they should think about inputs and outputs, but they do not know how to turn that into code.

So here is a more direct way to think about it.

11.4.1 If your output is a single number

It is probably a regression task.

That usually suggests:

  • output dimension of 1
  • floating-point labels
  • a loss like MSELoss

11.4.2 If your output is one of two classes

It is probably a binary classification task.

That usually suggests:

  • output dimension of 1
  • a loss like BCEWithLogitsLoss
  • turning the output into a probability or class prediction afterward

11.4.3 If your output is one of many classes

It is probably a multiclass classification task.

That usually suggests:

  • an output dimension equal to the number of classes
  • a loss like CrossEntropyLoss
  • labels stored as class indices

11.5 How Might You Modify the Template?

11.5.1 For tabular binary classification

You would often change:

  • YourDataset: load tabular features and labels
  • YourModel: make the final output 1-dimensional
  • criterion: switch to BCEWithLogitsLoss

11.5.2 For multiclass image classification

You would often change:

  • YourDataset: use an image dataset
  • YourModel: use a CNN or a pretrained model
  • criterion: keep CrossEntropyLoss

11.5.3 For regression

You would often change:

  • the final output dimension to 1
  • the labels to floating-point values
  • the loss to MSELoss

11.6 One Step Further: How Do You Go from a Task Description to Code?

Suppose your future task sounds like one of these:

  • “I want to predict whether a user will churn from tabular features.”
  • “I want to sort my own fruit photos into four classes.”
  • “I want to predict tomorrow’s sales from historical values.”

Force yourself to write these five sentences first:

  1. What shape is my input?
  2. What type of output do I need?
  3. How should I prepare the labels?
  4. What is the simplest sensible first model?
  5. What are my loss function and metric?

If you can answer those five questions clearly, your code already has a direction.

11.7 What You Really Want to Carry Forward

Not one exact code snippet, but a decision framework:

  1. identify the task type
  2. define the input and output format
  3. match the model and loss function to the task
  4. plug everything into a training template

Once that framework becomes natural, new tasks stop feeling like they require starting from zero.

11.8 An Important Exercise

Pick one small task you genuinely care about and write only these four lines:

  1. What is my input?
  2. What is my output?
  3. What loss function will I probably use?
  4. What should the first version of my model be?

If you can do that on your own, you have already started using PyTorch for your own needs.

11.9 This Chapter Is Really the Destination of the Whole Book

Everything earlier in the book points here:

  • tensors and shapes help you describe inputs and outputs
  • autograd and training loops help you understand how learning happens
  • Dataset and DataLoader help you organize your own data
  • MLPs, CNNs, and transfer learning give you several practical modeling starting points
  • the full project chapters show how to turn those starting points into trainable workflows with validation, checkpoints, and inference

So by the end, the most important thing is not “how many APIs did I memorize?”

It is this:

“When I get a new problem, I know how to begin the first version of a PyTorch project.”
