15  Common Failure Modes and How to Recover

Earlier in the book we introduced debugging habits. This chapter goes one step further and organizes common failures into patterns you can recognize.

That matters because independence in PyTorch is not just about writing code. It is about recovering when the code does not behave the way you expected.

15.1 Failure 1: The Code Runs, but Loss Barely Moves

Common causes:

  • learning rate is too small
  • model output and loss function do not match
  • labels are in the wrong format
  • features are poorly scaled
  • the model is too weak for the task

Good first checks:

  1. verify shapes
  2. verify label dtype
  3. try a different learning rate
  4. inspect one batch manually

15.2 Failure 2: Loss Goes Down, but Validation Stays Weak

Common causes:

  • overfitting
  • train/validation mismatch
  • data leakage in preprocessing
  • metric does not reflect the actual task

Good first checks:

  1. compare train and validation behavior side by side
  2. make sure preprocessing is fit on training data only
  3. reduce model capacity or add regularization
  4. inspect mistakes, not just averages

15.3 Failure 3: Shape Errors Keep Appearing

Common causes:

  • wrong flatten size before a linear layer
  • labels have an extra dimension
  • output dimension does not match the loss
  • batching changed the shape in a way you forgot

Good first checks:

  1. print inputs.shape
  2. print outputs.shape
  3. print targets.shape
  4. print intermediate shapes inside forward()

15.4 Failure 4: The GPU Is Available, but Training Is Still Slow

Common causes:

  • batch size is too small
  • data loading is the real bottleneck
  • CPU preprocessing dominates runtime
  • too much logging slows the loop down

Good first checks:

  1. confirm model and data are on the same device
  2. increase batch size if memory allows
  3. reduce unnecessary prints
  4. separate data-loading issues from model-compute issues

15.5 Failure 5: The Model Looks Fine During Training, but You Cannot Use It Later

Common causes:

  • no checkpoint was saved
  • only the last model was kept
  • no inference script exists
  • class names or preprocessing steps were not preserved

Good first checks:

  1. save the best checkpoint during training
  2. write a minimal inference script immediately
  3. keep preprocessing consistent between training and inference

15.6 A Better Way to Think About Failures

When something breaks, try not to think:

  • “I must not be good at this.”

A better question is:

  • “Which part of the pipeline is most likely wrong?”

That shift matters.

Strong practitioners do not avoid mistakes. They learn to localize them faster.

15.7 The Recovery Loop You Should Practice

When a run fails, go through this sequence:

  1. reduce the problem to one batch
  2. verify shapes and dtypes
  3. verify the loss function and target format
  4. make sure train/eval modes are correct
  5. try a simpler baseline if needed
  6. only then increase complexity again

This recovery loop is one of the most practical skills in the entire book.
