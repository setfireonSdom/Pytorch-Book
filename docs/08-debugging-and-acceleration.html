9  Chapter 8 Debugging and Speeding Things Up

This chapter is not tied to one specific code file. Instead, it teaches you how to work with all the code that came before.

Because once you start building your own tasks, the first problems are usually not “I do not know how to write version one.”

They are more like:

  • I wrote it, but it does not run
  • it runs, but the loss does not go down
  • the loss goes down, but the results are still bad
  • training is too slow, or memory runs out

So the goal of this chapter is simple:

to help you build a reliable way to debug your own projects.

9.1 A Good General Rule

When something goes wrong, do not change the model first.

Check these first:

  1. Is the data correct?
  2. Are the shapes correct?
  3. Is the loss function appropriate?
  4. Is the training loop correct?

Many problems are not caused by a weak model. They come from broken fundamentals.

9.2 If the Loss Does Not Go Down, Check in This Order

9.2.1 Step 1: Inspect one batch

Start by printing:

print(inputs.shape)
print(targets.shape)
print(targets[:5])

You want to verify:

  • the inputs are what you think they are
  • the labels are what you think they are
  • the batch shape is sensible

A surprising number of problems show up right here.

9.2.2 Step 2: Inspect the model output

outputs = model(inputs)
print(outputs.shape)

Now verify:

  • the output dimensions match the task
  • the output format matches what the loss function expects

For example, in a standard multiclass task:

  • outputs are often [batch_size, num_classes]
  • labels are often [batch_size]

9.2.3 Step 3: Make sure the loss function fits the task

This is easy to get wrong.

For example:

  • regression often uses MSELoss
  • binary classification often uses BCEWithLogitsLoss
  • multiclass classification often uses CrossEntropyLoss

If the task and the loss do not match, the model can behave very strangely.

9.2.4 Step 4: Verify the training loop itself

At minimum, check that these lines are present and ordered correctly:

optimizer.zero_grad()
loss.backward()
optimizer.step()

9.2.5 Step 5: Check whether the learning rate is extreme

If the learning rate is too large, the loss may oscillate or diverge.

If it is too small, the loss may barely move.

A very practical debugging move is to try a few different orders of magnitude:

  • 1e-1
  • 1e-2
  • 1e-3
  • 1e-4

9.3 How to Debug Shape Errors Quickly

The fastest method is usually not staring at the error message. It is printing shapes at key points.

Especially these three:

print("inputs:", inputs.shape)
print("outputs:", outputs.shape)
print("targets:", targets.shape)

If you are working with a CNN, go one step further:

temporarily print intermediate shapes inside forward().

That is one of the fastest ways to find a dimension mismatch.

9.4 If Validation Performance Is Bad, What Should You Check?

If the code runs but results are poor, start with:

  1. Are the training and validation sets properly separated?
  2. Are you using model.train() during training?
  3. Are you using model.eval() during validation?
  4. Is the data preprocessing reasonable?
  5. Does the metric actually match the goal of the task?

Do not stare only at the loss.

Sometimes loss goes down while the model is still learning the wrong thing.

9.5 Basic GPU and Performance Habits

Once you start using a GPU, build at least these habits:

  1. Put the model and data on the same device
  2. Use torch.no_grad() during validation
  3. If memory runs out, reduce batch size first
  4. Make one batch work before scaling up

These are not fancy performance tricks. They are part of a stable development process.

9.6 Do Not Optimize Blindly When Training Is Slow

First ask where the bottleneck actually is:

  1. data loading
  2. model computation
  3. batch size too small to use the GPU well
  4. too much unnecessary printing

Optimization only helps once you know what you are optimizing.

9.7 A Very Practical Debugging Checklist

When you start your own task, run through this checklist:

  1. Can I load one batch correctly?
  2. Can I run one forward pass?
  3. Can I run one backward pass?
  4. Do output shape and label shape match?
  5. Does loss show at least some downward trend within a few dozen steps?
  6. Is validation performance better than random guessing?

If those six checks pass, your project is usually past the “completely broken” stage and ready for more meaningful iteration.
