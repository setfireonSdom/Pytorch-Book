6  Learning a Full Project Through MLP Classification

This chapter introduces the first file that really feels like a complete project: code/05_mlp_classifier.py.

If the earlier chapters were about gathering building blocks, this chapter is where those blocks begin to feel like a working system.

6.1 Break the Code into Modules First

This file has four main parts:

  1. MLPClassifier: defines the model
  2. evaluate(): defines validation logic
  3. the dataset and DataLoader
  4. the training loop in main()

That is already very close to the structure of a real training script.

6.2 Start by Reading the Model Class

class MLPClassifier(nn.Module):
    def __init__(self) -> None:
        super().__init__()
        self.network = nn.Sequential(
            nn.Flatten(),
            nn.Linear(28 * 28, 128),
            nn.ReLU(),
            nn.Linear(128, 10),
        )

The main idea to build here is simple:

defining a model means defining how the input data moves through layers and what kind of output comes out at the end.

6.2.1 nn.Flatten()

MNIST images are 2D images, but a Linear layer expects a 1D vector for each sample.

So the first step is to flatten the image.

6.2.2 nn.Linear(28 * 28, 128)

This says:

  • each image starts as 784 numbers
  • the hidden representation will have 128 features

6.2.3 nn.ReLU()

This adds nonlinearity, so the model is not just stacking linear operations.

6.2.4 nn.Linear(128, 10)

This produces 10 class scores, one for each digit from 0 to 9.

6.3 Why This File Is Worth Revisiting

This is the first example that pulls everything together:

  • tensors
  • batching
  • a model
  • a loss function
  • an optimizer
  • validation accuracy

When you can read this file comfortably, you have started moving from “learning parts” to “understanding a full training script.”

6.4 Why Is forward() So Short?

def forward(self, x: torch.Tensor) -> torch.Tensor:
    return self.network(x)

It is short because the structure has already been defined in nn.Sequential.

In more complex models, forward() gets longer, but the idea stays the same: it describes how data moves through the model.

6.5 Now Read the Data Section

train_dataset = datasets.MNIST(...)
test_dataset = datasets.MNIST(...)

train_loader = DataLoader(train_dataset, batch_size=128, shuffle=True)
test_loader = DataLoader(test_dataset, batch_size=256, shuffle=False)

Two important things happen here:

  1. the training and test sets are separated
  2. each gets its own DataLoader

Pay attention to a common pattern:

  • training data usually uses shuffle=True
  • test or validation data usually uses shuffle=False

6.6 Follow the Shape of the Data Through the Model

  1. The input image starts as [batch_size, 1, 28, 28]
  2. After Flatten(), it becomes [batch_size, 784]
  3. The hidden layer maps it into a learned feature space
  4. The final output becomes [batch_size, 10]

Those 10 values are not probabilities yet. They are logits: raw class scores.

criterion = nn.CrossEntropyLoss()
optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)

These two lines decide:

  • how the model is judged
  • how the parameters are updated

6.7 Why Use CrossEntropyLoss?

Because this is a multiclass classification task.

It typically expects:

  • model outputs of shape [batch_size, num_classes]
  • labels of shape [batch_size]
  • labels stored as class indices, not one-hot vectors

That is why reading classification code always means checking both output shape and label format.

6.8 Read the Training Loop Carefully

for images, labels in train_loader:
    images = images.to(device)
    labels = labels.to(device)

    optimizer.zero_grad()
    logits = model(images)
    loss = criterion(logits, labels)
    loss.backward()
    optimizer.step()

By now, you should notice something important:

this has almost exactly the same structure as the earlier training loop you saw with linear regression.

The main things that changed are:

  • the data
  • the model
  • the loss function

That is one of the most important lessons in learning PyTorch for your own work.

You do not need to learn a brand-new system every time. You learn how to swap pieces into a familiar training skeleton.

6.9 evaluate() Matters Too

A lot of beginners learn how to train, but not how to validate.

In real projects, validation matters just as much.

The key pieces here are:

  • model.eval()
  • torch.no_grad()
  • argmax(dim=1)

They solve three different problems:

  • switch the model into evaluation mode
  • turn off gradient computation
  • convert class scores into predicted class labels

6.10 A Good Order for Reading This Kind of Code

Whenever you read a classification script, try this order:

  1. What shape do the inputs have?
  2. How many dimensions does the final output have?
  3. Which loss function is used?
  4. What format are the labels in?
  5. How is accuracy computed?

6.11 Good Changes to Try Right Away

  1. Change the hidden size from 128 to 256
  2. Add another Linear + ReLU block
  3. Change the optimizer from Adam to SGD

That is when you start to feel how model design, optimization, and training results affect one another in practice.
