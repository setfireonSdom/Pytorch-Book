5  Chapter 4 Understanding Data Flow Through a Custom Dataset

This chapter is built around code/04_custom_dataset.py.

In the previous chapter, we learned what the training loop looks like. Now we need to answer a key question:

where does the loader in that training loop actually come from?

5.1 First, Look at the Structure

This file does two things:

  1. defines StudentScoreDataset
  2. wraps it in a DataLoader

The example is tiny, but it captures the core logic of how data enters a real PyTorch project.

5.2 Start with StudentScoreDataset

class StudentScoreDataset(Dataset):
    def __init__(self) -> None:
        self.features = ...
        self.labels = ...

The basic idea is very simple:

  • features store the inputs
  • labels store the targets

This time we do not use a built-in dataset. We create a tiny tabular dataset ourselves.

That makes it easier to focus on how Dataset works.

5.3 What Is a Dataset, Really?

From the code’s point of view, a dataset only needs to answer two questions:

  1. How many samples are there?
  2. If I give you an index, what sample do you return?

That is why a custom dataset implements:

  • __len__
  • __getitem__

This is not just syntax. It is the core design.

PyTorch does not care whether your data comes from a spreadsheet, an image folder, or a database. It only needs a consistent way to request one sample at a time.

5.4 Understand __len__ and __getitem__

def __len__(self) -> int:
    return len(self.features)

This tells PyTorch how many samples exist.

def __getitem__(self, index: int) -> tuple[torch.Tensor, torch.Tensor]:
    return self.features[index], self.labels[index]

This tells PyTorch how to fetch one sample and its label.

The key idea is:

a dataset is not responsible for training. Its job is to deliver samples on demand.

5.5 Then What Does DataLoader Do?

DataLoader(dataset, batch_size=2, shuffle=True) says:

  • do not feed one sample at a time
  • group samples into batches
  • shuffle them during training

That is why the later training loop can be written naturally as:

for inputs, labels in loader:
    ...

5.6 Finally, Look at What the Main Program Prints

for batch_idx, (inputs, labels) in enumerate(loader, start=1):
    print(f"batch {batch_idx}")
    print("inputs shape:", tuple(inputs.shape))
    print("labels shape:", tuple(labels.shape))

This is a habit worth keeping.

Do not guess what a batch looks like. Print it and inspect it.

5.7 The Most Important Habit from This Chapter

When you start building your own project, do not begin with the model.

Ask yourself first:

  1. What does one sample look like?
  2. What does one label look like?
  3. What shape will they have after batching?

If those three things are unclear, model training usually breaks very quickly.

5.8 The Main Ideas Behind the File

After reading this chapter, you should understand:

  1. Dataset defines individual samples.
  2. DataLoader organizes batched iteration.
  3. shuffle=True is usually used for training data, not validation data.
  4. Label shape and dtype must match the loss function you plan to use.

5.9 Exercises to Try Immediately

  1. Add more samples to StudentScoreDataset.
  2. Change batch_size=2 to 3 or 4.
  3. Observe how the batch shapes change.
